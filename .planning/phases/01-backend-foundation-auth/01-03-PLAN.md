**G. Custom Access Token Hook — `apps/api/src/db/migrations/custom_access_token_hook.sql`:**

Per RESEARCH.md Pattern 4:
```sql
create or replace function public.custom_access_token_hook(event jsonb)
returns jsonb
language plpgsql
as $$
declare
  claims jsonb;
  user_firm_id text;
  user_role text;
begin
  select fm.firm_id::text, fm.role into user_firm_id, user_role
  from public.firm_members fm
  where fm.user_id = (event->>'user_id')::uuid
    and fm.deleted_at is null
  limit 1;

  claims := event->'claims';

  if user_firm_id is not null then
    claims := jsonb_set(claims, '{app_metadata,firm_id}', to_jsonb(user_firm_id));
    claims := jsonb_set(claims, '{app_metadata,firm_role}', to_jsonb(user_role));
  end if;

  return jsonb_set(event, '{claims}', claims);
end;
$$;

grant execute on function public.custom_access_token_hook to supabase_auth_admin;
revoke execute on function public.custom_access_token_hook from authenticated, anon, public;
```

This is a SQL file that must be run against the Supabase database (via SQL editor or as a migration). The hook injects `firm_id` and `firm_role` into every JWT's `app_metadata`.

**CRITICAL per RESEARCH.md pitfall 1:** The function MUST also be registered in the Supabase Dashboard under Authentication > Hooks. Just creating the function is not enough. This is listed in user_setup.
