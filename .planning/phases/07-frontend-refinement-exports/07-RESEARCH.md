## Summary

Phase 7 is a frontend-heavy polish phase with one significant backend addition (public REST API). The frontend work splits across three concerns: (1) refactoring `deal-card.tsx` from collapsible sections to a tabbed layout, (2) polishing the landing page, and (3) wiring keyboard shortcuts. The backend adds a new public API layer distinct from the internal `/api/*` routes.

The most important architectural finding is a **zod version mismatch**: the API package uses `zod@^3.24.2` but `@hono/zod-openapi` v1.x requires `zod@^4.0.0`. The solution is to install `@hono/zod-openapi@0.18.4` (the last version with `zod: '3.*'` peer dep), or upgrade the API package to zod v4. Given that `@hono/zod-validator@0.4.3` already supports both `^3.25.0 || ^4.0.0`, upgrading to zod v4 in the API package is the cleaner path — but must be verified for Drizzle ORM schema compatibility. Both options are documented below.

The CSV export is straightforward: `papaparse` is already installed in the frontend. The REST API needs a new `api_keys` DB table (not present in current schema), a middleware to validate `x-api-key`, and a new route group mounted at `/v1/` on the Hono app.

**Primary recommendation:** Use `@radix-ui/react-tabs` for the tab component, reuse InboxSidePanel for deal event detail, deliver CSV export via a Next.js route handler using `Papa.unparse()`, and build the public API as an `OpenAPIHono` sub-app mounted at `/v1` with `@hono/zod-openapi@0.18.4` (avoids zod upgrade risk).

### Pattern 1: Radix UI Tabs (Controlled Mode)

**What:** Tab switching controlled by React state; keyboard shortcut handler updates the state via `setActiveTab`.
**When to use:** When keyboard shortcuts (1-5) must set the active tab from outside the component.

```typescript
// Source: https://www.radix-ui.com/primitives/docs/components/tabs
import * as Tabs from '@radix-ui/react-tabs';

const TABS = ['terms', 'events', 'spread-history', 'news-research', 'reg-litigation'] as const;
type TabId = typeof TABS[number];

// In deal-card.tsx:
const [activeTab, setActiveTab] = React.useState<TabId>('terms');

// Keyboard handler (1-5 maps to tabs):
React.useEffect(() => {
  const handleKeyDown = (e: KeyboardEvent) => {
    // Guard: don't fire if user is typing in an input
    if (e.target instanceof HTMLInputElement || e.target instanceof HTMLTextAreaElement) return;
    const tabIndex = parseInt(e.key) - 1;
    if (tabIndex >= 0 && tabIndex < TABS.length) {
      e.preventDefault();
      setActiveTab(TABS[tabIndex]);
    }
  };
  document.addEventListener('keydown', handleKeyDown);
  return () => document.removeEventListener('keydown', handleKeyDown);
}, []);

// JSX:
<Tabs.Root value={activeTab} onValueChange={(v) => setActiveTab(v as TabId)}>
  <Tabs.List className="flex border-b border-border">
    {TABS.map((tab, i) => (
      <Tabs.Trigger
        key={tab}
        value={tab}
        className="px-4 py-2 text-sm font-medium text-text-muted data-[state=active]:text-text-main data-[state=active]:border-b-2 data-[state=active]:border-primary-500"
      >
        {tabLabels[tab]}
      </Tabs.Trigger>
    ))}
  </Tabs.List>
  <Tabs.Content value="terms"><TermsTab ... /></Tabs.Content>
  {/* ... other tabs */}
</Tabs.Root>
```

### Pattern 2: Deal Event Side Panel (CourtListener style)

**What:** Reuse InboxSidePanel pattern within deal card. Event list in Events tab renders items; clicking or pressing Enter opens the panel.
**When to use:** Events tab — event row click or keyboard Enter fires `setSelectedEventId`.

```typescript
// In deal-card.tsx — state alongside activeTab:
const [selectedEventId, setSelectedEventId] = React.useState<string | null>(null);

// j/k navigation state:
const [focusedEventIndex, setFocusedEventIndex] = React.useState(0);

// Keyboard handler for j/k/Enter/Esc (only active when Events tab is active):
React.useEffect(() => {
  if (activeTab !== 'events') return;
  const handleKeyDown = (e: KeyboardEvent) => {
    if (e.target instanceof HTMLInputElement || e.target instanceof HTMLTextAreaElement) return;
    if (e.key === 'j') { e.preventDefault(); setFocusedEventIndex(i => Math.min(i + 1, events.length - 1)); }
    if (e.key === 'k') { e.preventDefault(); setFocusedEventIndex(i => Math.max(i - 1, 0)); }
    if (e.key === 'Enter') { e.preventDefault(); setSelectedEventId(events[focusedEventIndex]?.id ?? null); }
    if (e.key === 'Escape') { e.preventDefault(); setSelectedEventId(null); }
  };
  document.addEventListener('keydown', handleKeyDown);
  return () => document.removeEventListener('keydown', handleKeyDown);
}, [activeTab, events, focusedEventIndex]);

// Layout: flex with side panel sliding in from right
<div className="flex h-full overflow-hidden">
  <div className={`flex-1 overflow-y-auto transition-all ${selectedEventId ? 'mr-[400px]' : ''}`}>
    <EventsTab events={events} focusedIndex={focusedEventIndex} onSelect={setSelectedEventId} />
  </div>
  {selectedEventId && (
    <DealEventSidePanel eventId={selectedEventId} onClose={() => setSelectedEventId(null)} />
  )}
</div>
```

### Pattern 3: CSV Export via Next.js Route Handler

**What:** Server-side route handler that builds CSV string and returns it with download headers. Using a route handler (not client-side) keeps backend data access clean, but for a deal board snapshot where data is already on the client, client-side generation with `Papa.unparse()` is simpler and sufficient.

**Decision:** Use client-side export (data is already fetched). Matches existing `exportDealCSV` pattern in `deal-card.tsx`.

```typescript
// Source: Papa Parse docs + existing deal-card.tsx pattern
import Papa from 'papaparse';

function exportDealBoardCSV(deals: Deal[]) {
  const rows = deals.map(d => ({
    deal_name: `${d.acquirerSymbol}/${d.symbol}`,
    acquirer: d.acquirer,
    target: d.target,
    offer_price: d.pricePerShare,
    current_price: d.currentPrice,
    spread_pct: d.currentSpread?.toFixed(2),
    p_close: d.p_close_base,
    status: d.status,
    announced_date: d.announcementDate,
    outside_date: d.outsideDate,
    filing_count: d.filingCount ?? 0,
    litigation_count: d.litigationCount ?? 0,
    regulatory_status: d.regulatoryStatus ?? '',
  }));

  const csv = Papa.unparse(rows, { header: true, escapeFormulae: true });
  const blob = new Blob([csv], { type: 'text/csv;charset=utf-8;' });
  const url = URL.createObjectURL(blob);
  const a = document.createElement('a');
  a.href = url;
  a.download = `j16z-deals-${new Date().toISOString().split('T')[0]}.csv`;
  a.click();
  URL.revokeObjectURL(url);
}
```

### Pattern 4: Public REST API with OpenAPIHono

**What:** New `OpenAPIHono` sub-app mounted at `/v1` on the Hono root, separate from internal `/api/*` routes. API key auth via middleware that checks `x-api-key` header against `api_keys` table.

```typescript
// Source: @hono/zod-openapi README
// apps/api/src/routes/v1/index.ts
import { OpenAPIHono, createRoute, z } from '@hono/zod-openapi';
import { apiKeyAuthMiddleware } from '../../middleware/api-key-auth.js';

export const v1App = new OpenAPIHono();

// Register security scheme
v1App.openAPIRegistry.registerComponent('securitySchemes', 'ApiKey', {
  type: 'apiKey',
  name: 'x-api-key',
  in: 'header',
});

// Apply API key middleware to all v1 routes
v1App.use('/*', apiKeyAuthMiddleware);

// Mount v1 routes
v1App.route('/deals', dealsV1Routes);

// Serve OpenAPI spec
v1App.doc('/doc', {
  openapi: '3.0.0',
  info: { version: '1.0.0', title: 'J16Z Public API' },
});
```

```typescript
// apps/api/src/index.ts — mount v1 alongside existing /api
app.route('/v1', v1App);

// Serve Scalar at /docs
// Source: https://scalar.com/products/api-references/integrations/hono
import { Scalar } from '@scalar/hono-api-reference';
app.get('/docs', Scalar({ url: '/v1/doc' }));
```

```typescript
// apps/api/src/middleware/api-key-auth.ts
import type { MiddlewareHandler } from 'hono';
import { adminDb } from '../db/index.js';
import * as schema from '../db/schema.js';
import { eq } from 'drizzle-orm';

export const apiKeyAuthMiddleware: MiddlewareHandler = async (c, next) => {
  const key = c.req.header('x-api-key');
  if (!key?.startsWith('sk_live_')) {
    return c.json({ error: 'Unauthorized' }, 401);
  }
  const [apiKey] = await adminDb
    .select()
    .from(schema.apiKeys)
    .where(eq(schema.apiKeys.keyHash, hashKey(key)))
    .limit(1);
  if (!apiKey) return c.json({ error: 'Unauthorized' }, 401);
  c.set('firmId', apiKey.firmId);
  await next();
};
```

### Pattern 5: api_keys Table Schema

**What:** Store hashed API keys (never plaintext) in a new table. The `sk_live_` prefix is shown once on creation; only the hash is stored.

```typescript
// apps/api/src/db/schema.ts — add:
export const apiKeys = pgTable('api_keys', {
  id: uuid('id').primaryKey().defaultRandom(),
  firmId: uuid('firm_id').notNull().references(() => firms.id, { onDelete: 'cascade' }),
  name: text('name').notNull(),
  keyHash: text('key_hash').notNull().unique(), // SHA-256 of sk_live_xxx
  lastUsedAt: timestamp('last_used_at', { withTimezone: true }),
  ...timestamps,
});
```

### Anti-Patterns to Avoid

- **Storing raw API key:** Only store `SHA-256(key)`. Return plaintext once on creation, never again. This matches the existing `api-keys-tab.tsx` UI pattern ("key shown once").
- **Using `@hono/zod-openapi@1.x` with zod v3:** v1.x requires zod v4 as peer dep. It will install but validation will silently fail or throw at runtime. Pin to `0.18.4`.
- **Putting public API routes on the internal `dealsRoutes` Hono app:** The internal routes use `firmContextMiddleware` + Supabase JWT auth. Public routes need a completely separate auth path (x-api-key).
- **Tab keyboard shortcuts firing in text inputs:** Always guard with `if (e.target instanceof HTMLInputElement || e.target instanceof HTMLTextAreaElement) return;`
- **No `URL.revokeObjectURL()` after CSV download:** Object URLs are not garbage-collected automatically. Call `revokeObjectURL()` after triggering the download.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Accessible tab component | Custom div with onClick | `@radix-ui/react-tabs` | Built-in keyboard nav (arrow keys, Home/End), aria-selected, focus management — all required for a11y |
| CSV serialization | Manual string join with commas | `Papa.unparse()` (already installed) | Handles quoting, newlines in values, formula injection, and character escaping |
| OpenAPI spec generation | Maintain a YAML/JSON file | `@hono/zod-openapi` createRoute + app.doc() | Spec stays in sync with route schemas automatically; no drift |
| API docs UI | Custom HTML page with spec | `@scalar/hono-api-reference` | Scalar is the established modern standard; 3 lines of code |

**Key insight:** The biggest trap in this phase is hand-rolling tab keyboard navigation. `@radix-ui/react-tabs` already handles arrow keys, Home/End, and ARIA — the custom 1-5 shortcut is additive on top, not a replacement.

## Common Pitfalls

### Pitfall 1: Zod Version Mismatch with @hono/zod-openapi

**What goes wrong:** Installing `@hono/zod-openapi@^1.x` in `apps/api/` (which uses zod v3) causes a peer dependency conflict. npm/pnpm may install it anyway but zod v4 schemas differ in API (e.g., `z.string().openapi()` method differs).
**Why it happens:** npm published a major version bump to 1.x requiring zod v4, not clearly communicated in package name.
**How to avoid:** Install `@hono/zod-openapi@0.18.4` explicitly. Verify with `pnpm why zod` in the api package to confirm only one zod version is resolved.
**Warning signs:** Runtime error `z.string().openapi is not a function` or TypeScript error on `.openapi()` chaining.

### Pitfall 2: Tab Content Flash on First Load

**What goes wrong:** Tab content for non-active tabs still renders in the DOM (Radix Tabs renders all content, but hides non-active panels via CSS). Async data loading in hidden tabs triggers network requests on mount.
**Why it happens:** Radix `<Tabs.Content>` renders all children immediately by default.
**How to avoid:** Lazy-load heavy tab content with conditional rendering inside `<Tabs.Content>` — check `activeTab === 'spread-history'` before mounting SpreadChart (recharts is expensive to mount). Use `forceMount={false}` (the Radix default) which unmounts inactive tabs.
**Warning signs:** Network waterfall shows 5 parallel requests when deal card opens.

### Pitfall 3: Keyboard Shortcut Conflicts Between Tabs and Global Nav

**What goes wrong:** The `1` key on the deal card tab shortcut conflicts with the Inbox filter shortcut `1` (toggle HIGH filter) defined in `inbox-timeline.tsx`.
**Why it happens:** Both attach to `document.addEventListener('keydown')` globally. When user navigates to a deal card from inbox, the inbox listener may still be active if inbox timeline component is in the layout.
**How to avoid:** The inbox keyboard shortcuts are mounted inside `inbox-timeline.tsx` which unmounts when the user navigates away to `/app/deals/[id]`. Next.js App Router unmounts the inbox page when navigating, so there is no conflict — BUT verify that `app-layout.tsx` does not mount the inbox timeline in the shared layout shell.
**Warning signs:** Pressing `1` on the deal card also triggers an unexpected filter update.

### Pitfall 4: API Key Never Hashed — Stored Plaintext

**What goes wrong:** Developer stores `sk_live_xxx` directly in the `api_keys` table. A DB read gives attacker full keys.
**Why it happens:** It's the simplest implementation; easy to overlook security requirement.
**How to avoid:** Use `crypto.createHash('sha256').update(rawKey).digest('hex')` before storing. On auth, hash the incoming `x-api-key` header and compare to stored hash.
**Warning signs:** `key_hash` column contains strings starting with `sk_live_`.

### Pitfall 5: Sidebar Layout Breaks Sticky Header

**What goes wrong:** When the event detail sidebar slides in on the Events tab, the sticky deal metrics header may stop being sticky if the container's `overflow` property is wrong.
**Why it happens:** `position: sticky` requires the ancestor not to have `overflow: hidden` or `overflow: auto`. The side panel implementation may wrap content in an `overflow-hidden` div.
**How to avoid:** The sticky header must be positioned outside and above the flex container that holds tab content + side panel. Structure: `<sticky header> → <flex row: tab content | side panel>`. The flex row should be `overflow-hidden` (to clip the sliding panel), not the overall page.
**Warning signs:** Deal metrics header scrolls away with tab content when side panel is open.

### Pitfall 6: Landing Page `tw-animate-css` vs Custom Keyframes Conflict

**What goes wrong:** `tw-animate-css` defines `@keyframes scroll`, `aurora`, `float` etc. Adding custom keyframes with the same names in a component creates cascade conflicts.
**Why it happens:** `globals.css` already imports `tw-animate-css` and defines `--animate-aurora`, `--animate-float`, `--animate-scroll`. Adding a CSS-in-JS or inline `<style>` tag with overlapping keyframe names creates undefined behavior.
**How to avoid:** Use only the existing `animate-` utilities from `globals.css` (`animate-float`, `animate-pulse-slow`). Do not add new keyframe definitions.

### @hono/zod-openapi Route Definition

```typescript
// Source: honojs/middleware README — @hono/zod-openapi@0.18.4 (zod v3)
import { z, createRoute, OpenAPIHono } from '@hono/zod-openapi';

const DealSchema = z.object({
  id: z.string().uuid().openapi({ example: 'abc-123' }),
  acquirer: z.string().openapi({ example: 'Microsoft Corp' }),
  status: z.string().openapi({ example: 'REGULATORY_REVIEW' }),
}).openapi('Deal');

const listDealsRoute = createRoute({
  method: 'get',
  path: '/deals',
  security: [{ ApiKey: [] }],
  responses: {
    200: {
      content: { 'application/json': { schema: z.array(DealSchema) } },
      description: 'List all deals for the authenticated firm',
    },
    401: {
      content: { 'application/json': { schema: z.object({ error: z.string() }) } },
      description: 'Unauthorized',
    },
  },
});

const v1App = new OpenAPIHono();
```

## Open Questions

1. **API keys table backend wiring**
   - What we know: `api-keys-tab.tsx` exists in frontend (mock UI) but there is no `api_keys` table in the DB schema and no `/api/api-keys` backend route.
   - What's unclear: Does the plan intend to wire up the existing mock UI to a real backend as part of this phase, or should the export route create keys through a minimal new endpoint?
   - Recommendation: Create the `api_keys` DB table + minimal CRUD routes (`POST /api/api-keys`, `DELETE /api/api-keys/:id`) as part of Plan 07-02 (REST API plan). The frontend mock UI in `api-keys-tab.tsx` already has the shape needed.

2. **Hono app architecture for `/v1` vs `/api`**
   - What we know: Current `apps/api/src/index.ts` mounts all routes under `/api`. The public API needs a separate auth path.
   - What's unclear: Whether to mount `v1App` on the existing root Hono instance or create a new HTTP server.
   - Recommendation: Mount on same root Hono instance: `rootApp.route('/v1', v1App)`. Keeps a single server. The v1 sub-app has its own middleware (api-key auth), so there's no JWT middleware leakage.

3. **Landing page "live product screenshots" — static images vs rendered components**
   - What we know: CONTEXT.md says "live product screenshots/mockups — not animated demos".
   - What's unclear: Whether these should be actual `<img>` screenshots or rendered React component mockups (like the existing `PolymarketCard`, `SearchResultsCard`, `SynthesisTerminal` components in `landing-page.tsx`).
   - Recommendation: Extend the existing React component mockup pattern (already in `landing-page.tsx`). Static `<img>` screenshots require a screenshot workflow; React mockups are maintainable and look intentional. This is Claude's Discretion per CONTEXT.md.
